# Synthetic User Testing Platform — Master Plan

# What This Is

We are building a **general-purpose synthetic user testing platform**. Any team can point it at their web product, define what they want to test, and receive a human-readable friction report generated by AI agents that behaved like real users.

The platform is product-agnostic. CoGrader is the first product it will be tested against, but the system is designed to work for any web application.

The four core capabilities:

1. **Persona generation** — realistic human users grounded in real behavioral evidence, with psychological depth, concrete constraints, and failure modes
2. **Plan generation** — naturalistic usage plans with concrete missions and realistic user states, not generic product tours
3. **Browser agent execution** — AI agents that run those plans against a live web application and produce inspectable traces, screenshots, and transcripts
4. **Scoring and evaluation** — structured metrics per session, plus a human-readable friction report

**How this differs from existing tools:**

- Not like Octomind or Magnitude — those are QA-focused. We generate behavior to uncover UX problems.
- Not like [SyntheticUsers.com](http://SyntheticUsers.com) — that is interview and research simulation, not browser-based interaction.
- Not like Jina's synthetic users — positioned around autonomous QA and CI/CD. We are focused on behavioral realism and UX insight.
- We are the combination: personas grounded in real evidence + naturalistic usage plans + actual browser agents + structured scoring.

# Architecture Overview

The system has five layers. Each is independently replaceable.

| Layer | What it does |
| --- | --- |
| 1. Persona Layer | Structured profiles grounded in real user evidence |
| 2. Task Planner Layer | Concrete missions + user state simulation + scenario variation |
| 3. Browser Operator Layer | Quick Mode (DOM) or Advanced Mode (OS-level computer vision) |
| 4. Scoring / Critic Layer | Structured metrics, trace grading, cross-session aggregation |
| 5. Report Layer | Human-readable friction reports per session and per cohort |

| Frontend | Full web UI |  |

## Why Browserbase + Stagehand

Running browser agents locally works for a prototype but becomes brittle at scale. Browserbase provides managed cloud browser infrastructure. Stagehand is the AI-native control layer built on top — it exposes `act()`, `extract()`, and `observe()` primitives designed for LLM-driven interaction. Together they give you real UI interaction, inspectable traces, and no browser infrastructure to operate.

Browser Use Cloud is a valid alternative if you prefer the browser-use Python library.

## Phase 2: Advanced Mode

When Advanced Mode is selected, the model layer switches to a full computer-use loop — the agent sees screenshots and clicks by pixel coordinates. All three frontier providers support this:

- **Anthropic Claude** — Computer Use API
- **OpenAI** — documented computer-use loops with operator-style agents
- **Google Gemini** — computer-use with Playwright as the client-side action handler

The user selects the provider. The execution harness is the same; only the model API call changes.

# Execution Modes

## Quick Mode (DOM-Based)

The agent interacts using Stagehand's AI-native primitives on top of Playwright.

- **Speed:** 5-15 minutes per session
- **Cost:** Low — a few cents of LLM API cost per session
- **Best for:** Rapid iteration, daytime testing, standard web UI
- **Limitations:** Struggles with cross-origin iframes, canvas-rendered elements, PDF viewers on external domains

## Advanced Mode (OS-Level / Computer Vision)

The agent sees screenshots and clicks by pixel coordinates. No awareness of the underlying HTML.

- **Speed:** 45-90 minutes per session
- **Cost:** Higher — multiple vision API calls per action
- **Best for:** Overnight runs, rich text editors, PDF viewers, canvas UI, third-party iframes
- **Limitations:** Slower, more expensive, occasionally misreads visual elements

**Recommended workflow:** Quick Mode during active development. Advanced Mode scheduled overnight for comprehensive coverage.

# Authentication

## Approach 1: Storage State Injection (Returning User Scenarios)

Playwright's `context.storageState()` captures the entire authenticated browser state — cookies, localStorage, and sessionStorage. Works with any auth method: SSO, Google OAuth, 2FA, anything.

**The flow:**

1. Platform launches a Playwright-controlled browser window
2. User logs in manually
3. Platform calls `context.storageState(path="session.json")` and saves the state
4. Each agent run starts with `browser.new_context(storage_state="session.json")` — already logged in

No passwords stored. The platform never handles credentials.

**Re-authentication frequency:**

| Session type | Typical duration | Re-auth needed |
| --- | --- | --- |
| Short-lived JWT (modern SaaS) | 1-24 hours | Once or twice a day |
| Persistent remember-me cookie | 7-30 days | Rarely |
| Session-only cookie | Single browser session | Every run |

The platform detects expiry automatically — if the agent's first navigation lands on a login page, it pauses and prompts re-authentication.

## Approach 2: Fresh Sign-Up (First-Time User Scenarios)

The agent navigates to the sign-up page, creates an account, handles email verification (using a service like Mailosaur or Mailinator), and starts as a brand new user.

**Best for:** Testing onboarding flows, first-time user experience, clean account state per run.

**Limitations:** Doesn't work for invite-only or paid products. Creates synthetic accounts in the user database — clean up regularly to avoid polluting analytics.

**The platform supports both.** The user selects per run:

- Fresh sign-up → for onboarding and first-time flows
- Session injection → for returning-user flows and gated products

# Onboarding Flow

## Step 1 — Product Input

The user provides one or more of:

- **A URL** (required minimum) — the platform crawls the product and infers its purpose, user types, and core workflows
- **Documents or files** (optional) — product spec, help docs, onboarding guide, marketing copy
- **A goals statement** (optional) — plain text describing what they want to test

The platform produces a product summary and shows it to the user before anything else happens.

## Step 2 — Persona Proposal and Confirmation

The platform proposes 4-6 personas as readable cards. The user can accept, edit, delete, or add personas.

**No run starts until the user confirms the persona set.** This is the first quality gate.

## Step 3 — Plan Proposal and Confirmation

For each confirmed persona, the platform generates 2-3 initial usage plans for review. The user can accept, edit, or regenerate.

**This is the second quality gate.** Once plans are confirmed, the user triggers the first run.

## Step 4 — Run and Report

Sessions execute in the background. The user monitors progress and receives friction reports when sessions complete.

# Making Personas Realistic

The biggest mistake in synthetic user testing is writing a colorful bio and calling it a persona. A paragraph of character description does not produce realistic agent behavior. **Structured behavioral fields do.**

## Ground Personas in Real Evidence

Whenever possible, persona fields should be derived from real sources:

- User interviews or sales call recordings
- Onboarding call notes
- Support tickets and help requests
- Demo objections
- Session recordings from real users
- Notes from pilots

The platform accepts these as uploads during setup and uses them to ground the generated personas. When no evidence is available, personas are generated from the product description alone and flagged as unvalidated.

## Structured Fields (Not Stories)

Key behavioral fields to capture:

- Years of experience and context
- Volume of work per week (e.g. essays, tasks, transactions)
- Tech comfort level
- Time pressure and patience budget
- Tolerance for AI autonomy
- Trust threshold for AI-suggested outputs
- Willingness to edit before accepting
- Retry willingness when something breaks
- Abandonment triggers

# Task Plan Generation

Plans should be **concrete missions**, not generic product tours.

## Concrete Mission Library (CoGrader Examples)

- Import an assignment from the LMS and confirm student submissions loaded
- Create a rubric from scratch for an argumentative essay
- Edit an existing rubric to add a missing criterion mid-grading
- Grade a batch of 10 essays, review AI feedback, and release to students
- Grade a borderline essay where the AI score seems too high
- Inspect and edit AI feedback on three essays before releasing any
- Compare two very similar essays and check AI consistency
- Override a grade the AI seems too confident about
- Handle a student submission that uploaded incorrectly
- Review class-level writing patterns after grading a full set

## Teacher State Simulation

Beyond the persona profile, each plan injects a specific **teacher state** — the situational context that affects behavior in this session:

- "You have 47 essays left and 22 minutes before your prep period ends."
- "You are skeptical of AI grading and will stop if any feedback feels generic."
- "You need comments you can send directly to students with minimal editing."
- "You tried to import from Google Classroom last week and it failed. You're approaching this with low confidence."

Teacher states are generated fresh for each run and logged alongside the session.

## Plan Variation System

Every plan is assigned a scenario dimension combination before generation to prevent repeated runs from covering the same ground.

| Dimension | Example Values |
| --- | --- |
| Mission | One from the concrete mission library |
| Entry point | Homepage, direct link, referral from email, LMS integration |
| Prior session state | First visit, returning with incomplete setup, returning with saved work |
| Time pressure | Relaxed, moderate, extreme |
| Distraction level | Focused, mildly distracted, frequently interrupted |
| Error encounter | No errors, one recoverable error, one unrecoverable error |

The plan generator receives the history of all previous scenario combinations for this persona and is instructed not to repeat any.

# Browser Agent Execution

## Behavioral Noise Injection

The system prompt makes the agent act imperfectly:

```
You are {name}. You are not a software tester. You are a real person
trying to get a task done during a busy week.

Current state: {teacher_state}

Behavioral rules:
- Do not read onboarding instructions or tooltips unless you are stuck
- Attempt the most visually obvious action first
- If something takes more than 2 clicks to find, try a different path
- If you see an error, try once to recover, then consider stopping
- Do not explore features unrelated to your current goal
- You may make mistakes and not immediately notice
- Express frustration in your action log when warranted
- Your patience budget is: {patience_budget from persona}
```

**Persona-conditional overrides** (applied on top of base rules):

| Persona type | Override |
| --- | --- |
| Carlos (compliant new user) | Read onboarding tooltips. Follow instructions before improvising. Ask for help after two failed attempts. |
| Maria (skeptic, low patience) | Skip all onboarding text. Go straight to the most obvious action. Abandon after one failure unless the task is almost done. |
| David (explorer, high patience) | Try non-obvious paths. Click on settings and secondary nav. Explore features adjacent to the current goal. |
| Linda (overloaded, rushing) | Skip everything skippable. Accept defaults. Never read more than one sentence of any message. Abandon if a flow takes more than 5 clicks. |
| James (reluctant, trust-focused) | Read explanatory text carefully. Hover on tooltips. Refuse to proceed if the AI rationale is not visible. |
| Priya (evaluator) | Test edge cases. Look for team/admin features. Check export and reporting flows even if not part of the mission. |

The plan generator selects the matching override block based on persona archetype and injects it into the system prompt.

## What the Agent Produces

Every session leaves behind inspectable artifacts:

- **Action trace** — timestamped log of every click, type, scroll, navigation, and error
- **Screenshots** — captured at key moments (page loads, errors, completion or abandonment)
- **Agent notes** — inline commentary on intent and frustration at each step
- **Session outcome** — goal achieved / partial / abandoned, with abandonment point logged

# Scoring Layer

## Per-Session Metrics

| Metric | What it measures |
| --- | --- |
| Task completion | Did the agent achieve the session goal? (binary + partial) |
| Time to first useful action | How long before the agent reached the core workflow? |
| Dead end count | How many times did the agent reach a state with no clear path forward? |
| Recovery count | How many times did the agent successfully recover from an error? |
| Help-seeking events | How many times did the agent look for docs, tooltips, or support? |
| Manual edits before acceptance | For feedback tasks: how many edits before the agent would release? |
| Confidence drops | Moments where the action log expressed uncertainty or frustration |
| Abandonment point | Where exactly did the session end without completion? |

## Product-Specific Metrics

Products can register additional metrics that are scored alongside the universal set. These are tagged as product-specific in the data model and excluded from cross-product comparisons.

**CoGrader example metrics:**

| Metric | What it measures |
| --- | --- |
| Grade-feedback contradiction | Cases where suggested grade and feedback text are inconsistent |
| Rubric-feedback alignment | Whether AI feedback references the rubric criteria the teacher defined |

## Cross-Session Aggregation

After a full cohort run:

- Which friction points appeared in 3+ sessions? (universal problems)
- Which appeared in only one persona? (persona-specific)
- Which missions had the lowest completion rates?
- What was the average abandonment point across the cohort?

---

# Output: Friction Report

One report per session, one summary per cohort run. Written for a product manager or designer, not a developer.

**Session report includes:**

- Scored metrics summary
- Critical friction points (severity 1-5)
- Mental model mismatches
- Missing or unfindable features
- Most likely churn reason
- Highest impact single change

# Data Model

The platform needs a clear data model from day one. Without it, repeated runs across product versions become an unmanageable collection of JSON files.

## Core Entities

**Product**

Represents a target web application being tested.

- `product_id` — unique identifier
- `name` — human-readable name
- `url` — base URL
- `crawl_summary` — inferred purpose, user types, and core workflows
- `created_at`, `updated_at`
- `versions` — array of tagged snapshots (e.g. `v1.2`, `post-onboarding-redesign`)

**Persona**

Reusable across products and runs.

- `persona_id`
- `product_id` — the product it was generated for (personas can be cloned across products)
- `name`, `role`, all structured behavioral fields
- `evidence_sources` — list of uploaded files or notes used to ground this persona
- `validated` — boolean, true if calibrated against a real user session

**Mission**

A concrete task from the mission library.

- `mission_id`
- `product_id`
- `description` — plain text task definition
- `expected_entry_point` — where the agent should start

**Plan**

A generated execution plan that connects a persona to a mission with specific scenario parameters. Plans are the unit of review in the confirmation step.

- `plan_id`
- `persona_id`
- `mission_id`
- `scenario_dimensions` — the dimension combination selected for this plan
- `teacher_state` — the situational context prompt to inject
- `steps` — ordered list of intended actions (high-level, not browser-level)
- `generated_at`
- `approved` — boolean, true after user confirms in the review step

**Run**

A single configured test execution (one or many sessions).

- `run_id`
- `product_id`, `product_version`
- `mode` — `quick` or `advanced`
- `auth_mode` — `session_injection` or `fresh_signup`
- `personas` — array of persona IDs included in this run
- `status` — `pending`, `running`, `complete`, `failed`
- `created_at`, `completed_at`

**Session**

A single agent execution — one persona, one plan, one browser interaction.

- `session_id`
- `run_id`, `persona_id`, `mission_id`
- `scenario_dimensions` — the dimension combination used
- `teacher_state` — the entropy parameter injected
- `mode` — inherited from run
- `status` — `pending`, `running`, `complete`, `abandoned`, `failed`
- `goal_achieved` — boolean + partial flag
- `abandonment_point` — step and reason if applicable
- `duration_seconds`
- `trace_path` — pointer to action trace file
- `screenshots` — array of screenshot paths
- `max_steps` — integer, safety cap on total actions (default 50 for quick mode, 200 for advanced mode)
- `hard_timeout_seconds` — integer, wall-clock limit (default 900 for quick mode, 5400 for advanced mode)

**Score**

Structured metrics attached to a completed session.

- `session_id`
- All per-session metrics (completion rate, dead ends, recovery count, etc.)
- `friction_events` — array of scored friction items
- `generated_at`

**Report**

Human-readable output generated from one session or one full run.

- `report_id`
- `session_id` or `run_id` (one or the other)
- `type` — `session` or `cohort`
- `content` — the report text
- `generated_at`

# Definition of Done for a Run

A run is not complete just because all sessions have finished executing. The following conditions must all be true before a run is marked `complete` and the cohort report is generated.

**Session completion criteria (per session):**

- Session reached a terminal state: `goal_achieved`, `abandoned`, or `max_steps_reached`
- Action trace has been written and is non-empty
- At least one screenshot was captured
- Score has been computed and attached

**Run completion criteria:**

- All sessions in the run have reached a terminal state
- No session is in `failed` status (a failed session — e.g. browser crash, auth expiry — triggers an alert and optionally a retry before the run can complete)
- Cross-session aggregation has run
- Cohort friction report has been generated

**What triggers the cohort report:**

The report generation job fires automatically when all sessions are complete and scored. If any session failed and was not retried, the report is still generated but flagged as partial, with the failed sessions noted.

**Session retry policy:**

A session that fails due to a recoverable error (auth expiry, browser crash, network timeout) is automatically retried once. A session that fails twice is marked `failed` and excluded from the cohort report. The user is notified.

# Synthetic Account Hygiene

When using fresh sign-up mode, the platform creates real accounts in the target product's database. Without a clear hygiene policy, this pollutes user analytics, inflates signup metrics, and creates noise in any ML models trained on user behavior.

## Policy

**Email convention:**

All synthetic accounts use a reserved email pattern:

```
synthetic+{run_id}_{persona_id}@{configured-domain}.com
```

Example: `synthetic+run_042_maria@testaccounts.yourproduct.com`

This makes synthetic accounts immediately identifiable and filterable in any analytics tool.

**Automatic exclusion:**

The platform generates a filter rule on first setup that can be applied in analytics tools (Mixpanel, PostHog, Amplitude, etc.):

```
email contains "synthetic+"
```

This rule should be added to the product's analytics configuration before the first run.

**Account cleanup:**

All synthetic accounts created during a run are logged in the `sessions` table. After a configurable retention period (default: 7 days), the platform prompts the user to delete them. Deletion can be manual (the user deletes from the product's admin panel) or automated if the product has an API that supports account deletion.

**Database tagging:**

If the target product supports user metadata, the platform sets a flag on each synthetic account:

```json
{ "synthetic": true, "run_id": "run_042", "created_by": "synthetic-testing-platform" }
```

This allows the product team to filter synthetic users at the database level, not just the analytics level.

**What to do before the first run:**

1. Configure the synthetic email domain in platform settings
2. Add the synthetic email filter to your analytics tool
3. Confirm your product does not send real emails to synthetic accounts (use a service like Mailosaur to catch and discard them)
4. Set the account retention period

# Limitations and Calibration

**Agents are more persistent than real users.** Tune abandonment thresholds in the behavioral noise prompt based on real session data, not instinct.

**Agents do not feel fatigue.** Simulate through teacher state prompts but cannot fully replicate cognitive load.

**Agents are not surprised.** Real users have contextual expectations built from years of prior experience. An agent's expressed frustration is linguistic, not experiential. Current research confirms that synthetic simulation gets shaky on diverse, high-variance human behavior, especially for subgroup modeling.

**Calibration strategy:** Run 2-3 real user sessions alongside the first AI cohort run. Compare friction points explicitly. Tune after every calibration round.

**The right mental model:** Use AI agents for volume and iteration. Use real users for ground truth. Neither replaces the other.
